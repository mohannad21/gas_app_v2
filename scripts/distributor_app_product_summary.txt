# Distributor App – Product Summary, Decisions, and Roadmap

## Purpose of this document

This document summarizes the full conversation and serves as a structured reference for:

* what the system is trying to become
* what has already been decided
* what still needs to be designed
* what should be built now, in the near future, and in the far future
* how to prioritize product and engineering work

It covers the distributor mobile app, the future customer/company apps at a high level, and the developer backoffice web app.

---

# 1. Overall product direction

The product is a **ledger-first distributor app** for managing gas-cylinder operations, customer activity, company activity, cash/bank movements, reminders, and later dashboard insights.

The system should remain based on these principles:

* the backend is the source of truth
* the ledger/reporting logic must be deterministic and auditable
* the daily report must reflect real posted events in correct order
* UI must be simple, fast, modern, and easy for daily operational use
* the app should support multiple workers with controlled permissions
* the system should be designed so it can later expand to:

  * a customer mobile app
  * a company web/mobile app
  * dashboard insights and forecasting

---

# 2. Main product areas

The product has been divided into these major areas:

## A. Distributor mobile app

Main operational app for the distributor and workers.

## B. Developer backoffice web app

Internal admin web app for the developer/operator to manage distributors, billing, plans, activation/deactivation, and support.

## C. Future customer app

Mobile app for end customers.

## D. Future company app

Web app, and maybe later mobile, for the company side.

---

# 3. Final mobile app navigation decision

The agreed navigation structure is:

## Bottom bar

* **Customers**
* **New**
* **Daily**
* **Dashboard**
* **Account**

This structure is preferred because it balances:

* daily operational speed
* discoverability
* low clutter
* separation between operational workflows and admin/personal settings

## Why this structure was chosen

* **Customers** is a real destination; it replaces the idea of a bottom tab called "Add Customer"
* **New** is the action/entry hub for posting new events
* **Daily** remains the daily report/timeline view
* **Dashboard** is separate from Daily because it is for trends, KPIs, and charts
* **Account** groups low-frequency personal/admin areas such as Profile, Settings, and Billing

# 4. What belongs under each bottom tab

## 4.1 Customers

This tab is for:

* customer list
* search
* filters
* balances and status indicators
* opening a customer’s timeline/activity view
* adding a new customer

### Important decision

A separate bottom tab called **Add Customer** is not needed. Instead:

* use **Customers** as the destination
* place a clear **Add Customer** action within that tab

### Customer-view direction

The customer profile should behave as a customer operational view, not as an orders-only screen.

Current decided direction:

* customer balances shown here should be **customer balances only**
* company balances should be removed from customer view
* the customer balances section should be **collapsed by default**
* tapping the balances section title should expand it

### Customer-view filter redesign direction

The current customer-view filters take too much space and should move toward progressive disclosure.

Preferred direction:

* fewer top-level filters visible at once
* selecting a top-level filter reveals a second level only when needed

Current examples of the preferred structure:

* **Money** -> All / Debt / Credit
* **12kg** -> All / Debt / Credit
* **48kg** -> All / Debt / Credit
* **Systems** -> All / Active / Inactive
* **Security check** -> Required / Not required
* if **Security check = Required**, a later extra level may be shown: All / Done / Undone

This is the current preferred direction, not a fully frozen final filter model.

## 4.2 New

This is the operational creation hub. It replaces the old idea of a simple Add screen.

Inside **New**, the app should expose these sections:

* **Customer Activities**
* **Company Activities**
* **Expenses**
* **Ledger Adjustments**

### Data-entry page cleanup direction

For the four operational sections:

* **Customer Activities**
* **Company Activities**
* **Expenses**
* **Ledger Adjustments**

the page should follow these product/UX cleanup rules:

* the filter area should sit directly between the main add button and the first list item
* there should be nothing else in between
* explanatory copy such as "Recent customer activities" or similar section-copy above the list should be removed
* the top-left **New** label on the data-entry page should be removed
* the tabs should be visually pushed upward
* the top-right 3-line menu / drawer should be removed
* everything attached to that menu should be removed with it
* dead code should not be left behind after that cleanup

### Filter rules for the New-page sections

**Customer Activities**

Should support:

* search by customer name
* activity-type filters

Important rule:

* customer-name search applies only to **Customer Activities**
* it does not apply to **Company Activities**, **Expenses**, or **Ledger Adjustments**

Current intended customer activity filter types should stay aligned with the customer-view activity model:

* Replacement
* Late Payment
* Return Empties
* Payout
* Sell Full
* Buy Empty
* Adjustments

**Company Activities**

* should have type filters only
* should not have customer-name search
* the exact final type-filter set still needs a final pass against the real company activity catalog

**Expenses**

* should have expense-type filters only
* should not have customer-name search
* the exact final expense filter set still needs final review

**Ledger Adjustments**

* should have adjustment-type filters only
* should not have customer-name search
* the exact final adjustment filter set still needs final review

### Customer Activities

Includes:

* Replacement
* Late Payment
* Return Empties
* Payout
* Sell Full
* Buy Empty

### Company Activities

This is broader than only refill. It can include company-side operational entries such as:

* Refill
* Company payment / company-related movements
* Buy / return related company entries
* other company-side actions already represented in the code

### Expenses

This is a separate section next to Customer Activities, Company Activities, and Ledger Adjustments.

Agreed structure for Expenses:

* **Expense**
* **Bank Transfer**

Inside this area, the user can:

* enter normal expenses
* move money from cash to bank
* move money from bank to cash

This means the expenses area should support two clear categories/tabs:

* **Expense**
* **Move Money / Bank Transfer**

### Ledger Adjustments

Includes correction-type entries such as:

* Cash Adjustment
* Inventory Adjustment

These are meant for controlled manual corrections and not as normal daily business actions.

## 4.3 Daily

This remains the day/timeline/report screen.

It should be used for:

* daily report
* chronological activity list
* expanding a day to see all activities
* quick daily totals
* reviewing the real sequence of events

### Important decision

**Dashboard is different from Daily.**
Daily is for operational review of what happened on a day.
Dashboard is for summary, trends, KPIs, and charts.

## 4.4 Dashboard

This is the analytics and summary area.

It should contain:

* KPI cards
* charts
* trends
* summary insights
* customer-oriented and business-oriented metrics

Charts should live here, not in Daily, not in New, and not in Profile/Settings.

## 4.5 Account

This groups low-frequency and admin/personal areas:

* **Profile**
* **Settings**
* **Billing**

This is preferred over having separate bottom tabs for Config and Profile.

# 5. Profile, Settings, and Billing

## 5.1 Profile

Profile is for **personal user settings only**.

Should include:

* name
* phone number
* optional email
* password change
* biometric unlock toggle
* language
* logout

### Not needed

* personal avatar is not needed

## 5.2 Settings

Settings is for **business configuration**, not personal settings.

Should include:

* prices
* system types
* reminder rules
* worker/team management
* permissions by role
* expense categories
* business settings
* low cash threshold / warning rules
* other distributor-level configuration

### Important clarification

Business config should not be inside Profile.

## 5.3 Billing

Billing is for the distributor’s subscription to the app.

Should include:

* current plan
* paid months
* amounts paid
* next due date
* current status
* later: receipts/invoices

This belongs under Account and should be visible only to the appropriate role, usually owner/admin.

# 7. Customer Activities view

Customer Activities are not a bottom tab; they live under Customers.

## Suggested event types in customer timeline

* Replacement
* Late Payment
* Return Empties
* Payout
* Sell Full
* Buy Empty
* Adjustment (if relevant)

### Customer activity scope

The customer activity view should stay focused on customer-facing operational events.

Include:

* Replacement
* Late Payment
* Return Empties
* Payout
* Sell Full
* Buy Empty
* Adjustment

Do not include:

* refill/company-side activity
* company-side activity that only indirectly touched the customer
* generic admin/system-maintenance events

### Customer activity card and expansion direction

Customer activities should use the same visual interaction pattern as Daily report activities as much as possible.

Direction:

* collapsed customer activity cards and collapsed Daily report cards should look and behave consistently where possible
* expanded customer activity cards and expanded Daily report cards should also feel visually related
* content may differ by context
* Daily expanded cards may include ledger-oriented boxes
* customer expanded cards may include balance-oriented boxes

The goal is one shared product language for:

* collapsed cards
* expanded cards

## Clarification of terms

### Adjustment

A manual correction event used when a normal business action does not fit, for example:

* correcting money balance
* correcting cylinder balance
* correcting a mistaken entry

### Customer balance wording direction

Customer balances should follow a split structure by:

* money
* 12kg
* 48kg

and by counterpart state:

* debt
* credit

Current intended structure:

* money debt / money credit
* 12kg debt / 12kg credit
* 48kg debt / 48kg credit

The exact final user-facing wording can still be refined later, but the structure should follow this split.

# 8. Company Activities and Ledger Adjustments

## Final decision

These do **not** need to exist as separate entries under Dashboard/More if they already live operationally under New.

The product direction is:

* keep them inside **New**
* allow creating entries there
* show recent entries there
* tap an entry for details/editing there

### Company balances placement

Company balances should not appear inside the customer view.

Instead:

* company balances should appear in **New -> Company Activities**
* they should sit below the **Add new company activity** button
* they should sit above the first data-entry item
* the company balances section should be **collapsed by default**

### Company balance wording direction

Company balances should follow the same structural split as customer balances:

* money
* 12kg
* 48kg

and by debt/credit-style counterpart buckets.

The exact final wording can still be refined later, but the structure should follow this split.

### Systems UI and negative-system rule direction

The Add / Update Systems UI is currently confusing and needs cleanup.

Agreed direction:

* systems UI should be simplified and made easier to understand
* if a system is in a negative state, no new activities should be added to it

The exact final technical definition of a "negative system" can still be finalized later if needed, but this is already agreed as the product rule direction.

This matches the actual workflow better.

### Why this decision was reached

Originally they were considered as possible secondary destinations, but the agreed direction is:

* if they are mainly entry + recent list + inspect/edit
* then they fit better under **New**
* and do not need to be duplicated as separate primary destinations elsewhere

# 9. Expenses and bank movement

One specific decision to complete now:

Under **New**, the app needs an **Expenses** section with two main parts:

* **Expense**
* **Move Money / Bank Transfer**

This should support:

* entering an expense
* moving money from cash to bank
* moving money from bank to cash

## Related business rules

* the ledger must track bank balance as well as cash
* the opposite of bank deposit must be possible
* if refill would cause negative cash, the app should warn the distributor and allow moving money from bank before saving
* bank transfer events must be visible in ledger/reporting

# 10. Profit calculation and pricing model

This is one of the most important missing architecture decisions.

## Agreed business rule

If prices change at the beginning of the month:

* sales start using the new **selling price** from the beginning of the new month
* but the system must still remember which cylinders came from which **buying price / refill cost layer**

So each sold cylinder conceptually needs both:

* the selling price at the time of sale
* the buying price of the inventory layer it came from

## Required data model direction

For each sale/replacement, the system should preserve:

* sale-time selling price
* quantity sold
* associated inventory/refill cost layer
* actual buying cost used for that quantity

## Recommended inventory costing model

Start with **FIFO**.

### Why FIFO

* easy to explain
* auditable
* deterministic
* answers the real business question of how much profit was made using cylinders bought at different costs

## Resulting future dashboard possibilities

This will later allow exact calculation of:

* revenue
* cost of goods sold
* gross profit
* profit by period/customer/gas type

---

# 11. Monthly price enforcement

The prices for buying gas from the company change at the first day of each month.

## Agreed requirement

The distributor must be forced to add the new monthly price before the first refill of the new month.

## Recommended implementation

This must be a **backend rule**, not only a frontend reminder.

If the current month’s required price is missing:

* refill creation should be blocked or strongly gated
* the user should be sent to the price setup flow

Workers should not be able to bypass this if permissions do not allow it.

# 12. System types and languages

## Requirement

When adding a customer, system types should appear by default.
The current experience is not liked.

## Multilingual consideration

System types should not rely only on translated display text.

## Recommended direction

Store:

* a stable internal key/code
* a display label
* later multilingual labels if needed

This allows:

* one stable system type identity internally
* different labels for different languages later

For now, the system can begin with one main business language and prepare the structure for translations later.

# 13. Reminders

Reminders became a major product area during the discussion.

## Core use cases

A reminder should be creatable when something is unresolved, for example:

* customer did not pay enough
* customer did not return enough empties
* follow-up expected in one week
* system/security follow-up needed
* any other distributor reminder workflow

## Example given

A customer takes full cylinders without returning empties because they need to cook, and promises to return the empties a week later. The distributor should be able to set a reminder date and receive a reminder in one week.

## Requirement

The reminder date field should appear only when relevant, for example when:

* payment is incomplete
* empty return is incomplete
* other follow-up is needed

## Reminder behavior in the app

The distributor must be able to:

* see what to collect and from whom
* tap quickly into the relevant action
* add the missing payment or empty return in one click/flow
* defer a reminder
* set a new date
* mark it as done
* cancel/remove it

## Product direction

Reminders should be a dedicated topic/feature, not just a simple date field.

## UI placement

Reminders should have their own bottom tab.

## Possible reminder categories

* overdue
* today
* upcoming
* recently done

# 14. Users, workers, roles, and permissions

## Main business requirement

The distributor can add or remove workers.

## Roles (initial suggestion)

* Owner
* Manager
* Worker / Driver

## Important permission decisions

### Worker/Driver should **not**:

* see buying prices
* see company buy cost information generally
* delete records freely
* change global config
* manage subscription/billing

### Worker/Driver should be able to:

* create daily customer activities
* edit within defined rules
* view operationally relevant information

## Important security concern

The worker must not be able to delete everything in case of conflict.

## Recommended rule

* workers can create
* workers can update under controlled rules
* deletes should be restricted to owner/manager or handled through audited reversal logic

# 15. Authentication and biometrics

## Distributor onboarding flow

When a new distributor is added:

* the distributor receives a link to install the app
* they receive temporary username and password
* they must change the password immediately on first login
* after that, the app asks them to activate biometric unlock / face recognition

## Login identifier discussion

Username can be:

* phone number or email
* with optional email support in profile

## Biometrics

Biometric unlock is desired as a convenience layer after login.

### Product direction

* first login with credentials
* force password change if temporary
* then prompt for biometric unlock

This should be implemented as app unlock/session unlock, not as the only identity method.

# 19. Developer backoffice web app

A simple web app is needed for the developer/operator.

## Main purpose

Manage distributors and the platform’s subscription/billing/admin operations.

## Required features

* add distributor
* remove/deactivate distributor
* reactivate distributor
* see billing/payment history
* suspend if not paid
* manage temporary credentials/invites
* resend invite
* reset password if needed
* view plan/status
* later possibly manage support details and feature flags

## Billing and plan control

The developer should be able to manage subscription plans and platform billing from the backoffice web app.

### This includes

* change subscription plans
* change plan prices
* manage who is active/suspended
* record which months were paid and how much
* record manual payments (for example when paid at the bar)

## Important distinction

There are two different types of prices:

### A. Platform subscription prices

What the distributor pays for using the app.
These are controlled by the developer in Backoffice.

### B. Business gas prices inside the distributor app

What the distributor uses operationally for gas buying/selling.
These are controlled inside the distributor’s Settings, not globally by the developer in normal cases.

---

# 20. Backoffice billing/pricing architecture direction

The developer should be able to change plans and subscription prices for the distributor app.

## Important design rule

Billing history should not be overwritten.

If plan price changes:

* old billing records remain as they were
* the new price applies from an effective date forward

This implies future need for structures such as:

* plan catalog
* plan price history
* distributor subscription
* billing records
* effective date logic

# 21. Screenshot/testing workflow improvement

A practical concern was raised: manually testing on phone, taking screenshots, sending them to WhatsApp, and uploading them again is too slow.

## Key conclusion

The UI cannot be extracted as final screenshots from code alone without rendering, but the app can be rendered on web and used for automatic screenshots.

## Product/process direction

* run the Expo app on web/browser
* create stable preview/demo routes with mock data
* use browser screenshots for fast UI review
* keep phone testing for final QA

This should significantly speed up UI review and issue detection.

# 26. Final agreed direction in one short summary

The distributor app should become a clean mobile product with this main structure:

* **Customers** for people and their activity
* **New** for posting operational events
* **Daily** for day-based report/timeline
* **Dashboard** for insights and charts
* **Account** for Profile, Settings, and Billing

Inside **New**, the main operational sections are:

* Customer Activities
* Company Activities
* Expenses
* Ledger Adjustments

Current UX direction also includes:

* cleaner New/data-entry screens with less chrome and no obsolete drawer/menu
* filters placed directly under the add button
* customer balances staying in customer view only
* company balances moving to Company Activities
* customer and Daily activity cards following one visual interaction language as much as possible

The system must support:

* exact, auditable ledger logic
* multiple workers with restricted permissions
* reminders as a real feature
* bank and cash tracking
* monthly pricing enforcement
* future exact profitability through cost layers/FIFO
* a separate developer backoffice web app for distributors, subscription plans, billing, and activation/deactivation

This is the current reference direction for planning and prioritization.
