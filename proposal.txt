1) Executive Summary

This project is a lightweight, distributor-first operations system for managing LPG cylinder business workflows end-to-end: customer orders, cylinder exchanges, inventory movements, company refills and settlements, cash flow, and day-by-day reporting. The foundation remains ledger-first accounting: every real-world action posts canonical ledger entries, and all balances are derived deterministically from that ledger.

Customer and company balances are shared relationship balances, not per-screen approximations. The backend remains the source of truth for reports and accounting state. The frontend still calculates some preview and display helpers today, but the long-term direction is to keep important accounting meaning aligned with backend truth.

Operational money terminology also needs one clear direction:
- Current
  - the ledger account still uses the term cash internally
  - the user-facing frontend term is Wallet for the distributor's on-hand money
  - the daily report top money value is the Wallet value from the ledger
- Next
  - keep Wallet wording consistent on new and touched frontend surfaces
- Later
  - remaining internal naming can be cleaned up only where it improves maintainability without changing accounting meaning

The wording in this document is target wording direction, not a statement that every visible label in the UI is already finalized. Final wording is still being cleaned up across customer wording, company wording, filter wording, and card wording.

The daily report wording in the frontend is currently the best active wording reference. It is treated as correct for the current phase, while still leaving room for future wording cleanup if product needs sharper phrasing later.

The daily report is intentionally a layered operational surface:
- Current Balances
- collapsed day cards
- event feed
- expanded event detail

Day detail is intentionally lazy-loaded on expand rather than eagerly prefetched. Performance and determinism are product requirements because a slow or inconsistent report directly reduces user trust.

Status framing used throughout this proposal:
- Current = already in the app now
- Next = being changed soon
- Later = future work after current cleanup tickets

2) Background & Problem Statement

Background

As a gas cylinder distributor, daily operations are a mix of physical inventory exchanges (full vs empty cylinders across 12kg and 48kg), customer settlements, company interactions, expenses, cash handling, and corrections. These actions happen continuously and often under time pressure. Small recording errors compound quickly: inventory counts drift, customer relationship balances become unclear, company balances become disputed, and cash movement becomes hard to reconcile.

Problem Statement

The current pain points are:

High-frequency transactions with low tolerance for mistakes
- Orders, exchanges, refills, payouts, late payments, adjustments, and returns happen throughout the day.
- The UI must be fast and error-resistant without sacrificing correctness.

Semantic confusion around relationship balances and sign conventions
- The underlying business model is signed internally.
- The user should not need to interpret raw sign semantics.
- The product therefore needs one visible wording contract that is applied consistently across screens.

Data inconsistency risks between client and server
- If the client computes accounting meaning independently, discrepancies appear, especially with edits, backdating, or identical timestamps.
- The backend must remain authoritative for accounting truth.

Edits and backdating can corrupt explanation even when ledger truth is correct
- Operational reality requires update and delete flows.
- If reports depend on stale snapshots instead of canonical ledger state at the event boundary, later wording becomes misleading.

Reconciliation must be possible without exposing complexity upfront
- The report surface must stay scan-first by default.
- Users must still be able to expand a day and then an event to see structured before/after impacts for audit and dispute resolution.

Performance and determinism are product requirements
- Day detail should load only when a day is expanded.
- Expanding one day should not trigger unrelated detail fetches, frontend rerender storms, or backend per-event customer ledger aggregation explosions.

3) Goals & Non-Goals

Goals

Ledger-first correctness
- Every action posts ledger entries as the source of truth.
- All balances and before/after states are derived from the ledger.
- Deterministic ordering and tie-breakers prevent nondeterministic snapshots.

Fast and user-friendly daily operations
- Recording an event should be quick and require minimal fields.
- The UI should hide irrelevant zeros and present only actionable information.
- The report should remain scan-first with details on demand.

Shared wording and explanation consistency
- The same balance fact must be rendered the same way across preview, balances, day cards, event cards, and expanded detail.
- Cross-zero transitions must be explicit and understandable.

Safe editing and backdating
- Update flows preserve intended happened_at/day semantics.
- Changes remain auditable through reversal plus repost or equivalent append-only logic.

Performance architecture as a requirement
- The daily report must fetch and render lazily.
- Day-detail generation must avoid confirmed N+1 patterns and unnecessary whole-table loads.

Non-Goals (for this phase)
- Full accounting/invoicing system (tax/VAT, formal receivables aging, full general ledger UX)
- Multi-warehouse / multi-driver fleet optimization
- Online payment gateways unless explicitly added later
- Complex ERP integrations unless later required
- Forecasting / ML beyond operational reporting

5) Domain Model & Source-of-Truth (Core Accounting Semantics)

5.1 Core Principle: Ledger-First, Server-Authoritative

The ledger is the single source of truth. Every business action must post one or more ledger entries that represent canonical deltas. All derived states—inventory, cash, bank, customer balances, company balances, report transitions, and before/after panels—are computed from the ledger, not from UI-calculated snapshots.

Why this matters
- Prevents client/server drift.
- Makes edits auditable.
- Enables deterministic reporting at exact event boundaries.

5.2 Primary Domain Entities (Conceptual)

Event / Transaction (Source)
- Represents a user action (order, collection, refill, adjustment, expense, init, etc.).
- Has id, event_type, happened_at, day, created_at, note, created_by, is_reversed, reversed_id or equivalent.

LedgerEntry
- Represents one atomic delta to one account/unit.
- Has id, source_type, source_id, account, unit, gas_type?, state?, delta, happened_at, day, created_at.

Derived Report Views
- Current balances
- day_v2 event feed
- daily_v2 day cards
- expanded before/after event detail

5.3 Canonical Balance Model

Customer <-> distributor
- one shared money balance per customer
- one shared 12kg cylinder balance per customer
- one shared 48kg cylinder balance per customer
- balances are customer-level, not system-level
- system is operational context for some events, not a balance dimension
- customer detail is therefore a customer operational profile, not a system-balance view

Customer summary metrics
- Current
  - the business wants cylinder counts by gas type to be the visible customer summary, not number of orders
  - Cylinders Ordered is a customer-level lifetime summary, not a count of event rows
  - Cylinders Ordered must be tracked separately for 12kg and 48kg
- Next
  - include:
    - Replacement installed cylinders
    - Sell Full cylinders
  - do not include:
    - late payment / payment-only events
    - return empties
    - buy empty
    - adjustments
- Later
  - these 12kg / 48kg totals can feed dashboard and statistics views more directly than raw order counts

Company <-> distributor
- one shared company money balance
- one shared company 12kg cylinder balance
- one shared company 48kg cylinder balance

Operational
- full 12kg inventory
- empty 12kg inventory
- full 48kg inventory
- empty 48kg inventory
- cash in the ledger, shown as Wallet in user-facing frontend wording
- bank if used

5.3.1 Wallet / Bank Operational Rule

Current
- the top money value in the daily report comes from the ledger cash balance
- Wallet means the distributor's on-hand money tracked in the ledger
- Wallet to Bank means moving money from ledger cash into bank
- Bank to Wallet means moving money from bank into ledger cash
- the transfer UI has exactly two fixed tabs:
  - Wallet to Bank
  - Bank to Wallet
- there is no direction selector inside the transfer UI
- Wallet to Bank is limited by the current Wallet amount
- Bank to Wallet can add however much the distributor chooses to move in

5.4 Sign Meaning (Canonical Internal Model)

Customer
- positive money = customer owes distributor
- negative money = distributor owes customer
- positive cylinder = customer owes empties
- negative cylinder = distributor owes full cylinders

Company
- positive money = distributor owes company
- negative money = company owes distributor
- positive company cylinder = company owes full cylinders
- negative company cylinder = distributor owes empty cylinders to company

Operational
- positive inventory/cash/bank = distributor has more on hand
- negative inventory/cash/bank = deficit/overdrawn condition

The UI wording is derived from this model, but users do not see raw sign semantics.

5.5 Deterministic As-Of Boundaries

Ordering key for correctness and boundaries:
- happened_at ascending
- created_at ascending
- id ascending

Snapshot computation must accept a full boundary, not happened_at alone. This guarantees:
- stable before/after even when timestamps collide
- stable grouped-event boundaries
- deterministic report wording across environments

5.6 Edit Semantics: Reverse + Repost

Edits must not rewrite history implicitly.

Rule
- update = reverse old effect + create new effect
- delete = reverse old effect
- both preserve original business time unless the user explicitly changes the effective time

This keeps historical reporting accurate while preserving auditability.

5.7 Report View Contract

The reporting layer is not a raw ledger browser. It is a structured explanation layer built from canonical ledger truth.

Each event in day_v2 should provide enough structured fields to render:
- a collapsed event row
- a transition-oriented wording block
- an expanded before/after delta panel

Every user-facing event row in the report feed is expandable.

6) Core Workflows (End-to-End)

Below is the business workflow view from user action -> backend posting -> derived effects -> collapsed emphasis -> expanded detail.

6.1 Customer-side Events

Customer detail timeline scope
- the main customer activity view on customer detail is customer-facing operational history, not a generic system/admin timeline
- it should include:
  - Replacement
  - Late payment
  - Return empties
  - Payout
  - Sell Full
  - Buy Empty
  - Adjustments
- it should not include:
  - refill / company-side activity
  - company-side activity that only indirectly touched the customer
  - generic admin or system-maintenance events

System binding rules
- balances remain customer-level
- systems remain operational context, not separate balance owners
- only these customer activity types may be tied to a system:
  - Replacement
  - Sell Full
- Sell Full may:
  - be tied to an existing system
  - create a new system and be tied to that new system
  - remain not tied to a system
- this allows existing-system delivery, new setup / new house delivery, and one-off delivery without forcing false system ownership

Replacement
- Scope: customer + operational
- Operational meaning: deliver full cylinders, receive empties, optionally collect payment
- Changes:
  - customer money
  - customer 12kg or 48kg relationship balance
  - inventory
  - cash
- Collapsed UI emphasizes:
  - installed / received / paid
  - resulting customer state if unresolved or cross-zero
- Expanded detail shows:
  - customer money before/after
  - customer gas balance before/after
  - cash before/after
  - relevant inventory before/after

Sell Full
- Scope: customer + operational
- Operational meaning: sell full cylinders without empty-return semantics
- Changes:
  - customer money
  - inventory full
  - cash
- Collapsed UI emphasizes sold quantity, paid, and resulting money state
- Expanded detail shows customer money, cash, and relevant full-inventory deltas

Buy Empty
- Scope: customer + operational
- Operational meaning: buy empty shells from the customer
- Changes:
  - customer money
  - empty inventory
  - cash
- Collapsed UI emphasizes bought quantity, paid, and resulting money state
- Expanded detail shows customer money, cash, and relevant empty-inventory deltas

Payment / Late Pay
- Scope: customer + operational
- Operational meaning: customer pays money later
- Changes:
  - customer money
  - cash
- Collapsed UI emphasizes the resulting money state
- Expanded detail shows customer money before/after and cash before/after

Payout to Customer
- Scope: customer + operational
- Operational meaning: distributor pays customer, usually reducing or clearing customer credit
- Changes:
  - customer money
  - cash
- Collapsed UI emphasizes what happened to prior customer credit
- Expanded detail shows customer money before/after and cash before/after

Return Empties / Late Return
- Scope: customer + operational
- Operational meaning: customer returns empties outside an order flow
- Changes:
  - customer 12kg and/or 48kg relationship balance
  - empty inventory
- Collapsed UI emphasizes resulting cylinder state
- Expanded detail shows customer cylinder before/after and inventory empty before/after

Customer Adjustment
- Scope: customer
- Operational meaning: explicit manual correction of customer money and/or cylinder relationship balances
- Changes:
  - customer money
  - customer 12kg
  - customer 48kg
- Collapsed UI emphasizes who was adjusted and the resulting state
- Expanded detail shows the relationship balances that changed

7.1 UX Principles (Global)

Scan-first, details on demand
- The default report surface must be scannable in seconds.
- Expansion reveals structured accounting truth, not a raw ledger line-item dump.

One visible wording system
- Current-state screens use current-state wording.
- Transition screens use transition wording.

Hide zeros and redundancy

Deterministic ordering
- Users must trust that before/after is correct.

8) Data Integrity & Consistency Strategy

8.1 Invariants (Must Always Hold)

Ledger is authoritative
- UI snapshots are derived.
- Backend recalculates after posting.

Deterministic boundaries
- Any snapshot taken after event X must use boundary = (happened_at, created_at, id), not happened_at alone.

Reports do not trust stale source snapshots for truth
- Stored debt_* style fields may exist as optional cached fields.
- Reports must derive visible event-boundary truth from the canonical ledger.

Grouped events remain logically grouped
- Grouped returns and grouped customer adjustments should render as one logical event where product semantics intend one user action.

Init/opening state remains auditable
- Opening balances must not become invisible hidden ledger truth.

8.2 Update/Delete Consistency

Update
- preserve reversal + recreate behavior
- preserve or intentionally move business time
- recalculate report state from ledger boundaries

Delete
- reverse the event
- remove it from active report/feed
- ensure later report wording updates automatically because it is ledger-derived

9.4 Performance Architecture (Product Requirement)

Backend day_v2
- avoid per-customer-event ledger aggregate explosion
- seed involved customer states once before the day
- update those states in memory across the day’s sorted events
- load only involved customers, systems, and expense categories for day detail

Frontend reports
- do not auto-fetch latest day detail on screen open
- fetch day detail only when a day is expanded
- keep FlatList virtualization enabled
- memoize/split day rows so one expanded day does not rerender the whole list
- avoid unnecessary cache resets that retrigger duplicate day-detail fetches

10) API Contracts & Versioning

10.1 Contract Principles

Version report semantics explicitly
- day_v2 and daily_v2 are semantic contracts, not arbitrary payloads

Server returns structured truth
- frontend should render server-authored truth, not reconstruct accounting meaning independently

Do not silently overload fields with new meanings
- if semantics differ materially, use a new field or versioned change

10.2 Report Contracts

GET /reports/day_v2?date=YYYY-MM-DD

Returns:
- ordered events for that business day
- one logical row per user-facing action
- structured balance_transitions for visible wording
- relevant before/after fields for:
  - customer balances
  - company balances
  - cash
  - bank
  - inventory

GET /reports/daily_v2?range=...

Returns:
- day cards
- day-local summary math
- structured problem_transitions
- current balances section if the screen includes it, or enough canonical data to render it separately

- init / opening events

10.2.2 Daily Report Expandability (MUST)

Every event row in the report feed is expandable.

Expansion is not limited to cash + inventory. It can show any relevant relationship and operational before/after fields for that event.

10.2.3 Server/UI Responsibility Boundary (MUST)

Frontend
- renders current-state and transition wording using shared helpers
- may still compute preview/display helpers locally
- does not own authoritative report balances

Backend
- provides event ordering, balance transitions, and before/after fields
- remains the source of truth for accounting state

10.2.4 Determinism Semantics (MUST)

Authoritative ordering key for correctness:
- happened_at
- created_at
- id

Grouped events must define a stable boundary derived from ledger ordering so their before/after states are deterministic.

10.3 Data-Entry Preview Contract

Customer preview
- uses canonical customer balance data
- not list snapshots

Company preview
- uses canonical company balance data
- not day-report rows

Preview wording
- current-state wording before any entered delta
- transition wording for expected after-save state

Preview safety
- must not show false zero because there is no report row for today

10.4 Versioning Policy

If report wording semantics, expansion payload semantics, or required before/after fields change materially, version the contract explicitly instead of silently changing meaning.

10.5 Compatibility & Quality Tests

Contract tests should cover:
- event presence in day_v2
- structured balance_transitions / problem_transitions shape
- required before/after fields for supported event types
- deterministic ordering and grouped-event boundaries

11) Technical Approach & Architecture

11.1 Approach Summary

Event-sourced accounting remains the basis: business activities are source records, and the ledger is the canonical accounting record derived from them.

Server-authoritative reporting remains the rule: the backend constructs structured report truth. The frontend renders that truth using the shared transition formatter and current-state formatter.

11.2 Key Architecture Decisions

Single report screen model
- Current Balances + day cards + event feed + expanded detail exist in one coherent report experience

Server-authored structured report fields
- balance_transitions
- problem_transitions
- before/after fields for relationship and operational balances

Posting services remain the only ledger writers
- routers validate + call posting services

Edits use reverse + repost
- preserve business time semantics unless explicitly changed

11.3 Logical Components

Backend
- API surfaces
- posting logic
- ledger helpers
- report assembly
- source tables + ledger entries
- schemas and report contracts

Frontend
- typed API client
- schemas and types
- shared wording helpers

- data fetching and caching

ledger_entries

- same timestamp events -> correct deterministic before/after
- updates and deletes preserve historical reporting semantics
- grouped returns and grouped adjustments render correctly

Preview correctness
- customer preview uses canonical customer balances
- company preview uses canonical company balances
- preview wording matches saved result semantics

Wording consistency

Performance regression gates
