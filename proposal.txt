1) Executive Summary

This project is a lightweight, distributor-first operations system for managing LPG cylinder business workflows end-to-end: customer orders, cylinder exchanges, inventory movements, supplier (company) refills, cash flow, and day-by-day reporting. The core design principle is ledger-first accounting: every real-world action posts canonical ledger entries, and all “balances” (inventory, customer status, company status, cash) are derived deterministically from the ledger.

The product targets one outcome: fast daily operation with high correctness. Users should be able to record actions (replacement, sell full, buy empty, refills, payments, adjustments, expenses) in seconds, and later audit them via an expandable daily timeline that shows before/after impacts without overwhelming the user during scanning. The system is designed to minimize cognitive load in the field while preserving full traceability for reconciliation.

2) Background & Problem Statement
Background

As a gas cylinder distributor, daily operations are a mix of physical inventory exchanges (full vs empty cylinders across 12kg and 48kg), customer settlements, and supplier interactions. These actions happen continuously and often under time pressure. Small recording errors compound quickly: inventory counts drift, customer debts become unclear, and supplier balances become disputed. The result is time lost reconciling, money left uncollected, and operational uncertainty (“Do we actually have enough empties?” / “Who still needs to pay?”).

Problem Statement

The current pain points are:

High-frequency transactions with low tolerance for mistakes
Orders, exchanges, refills, late payments, and returns happen throughout the day. The UI must be extremely fast and error-resistant.

Semantic confusion around debts/credits and sign conventions
Cylinder “debt/credit” and cash “debt/credit” can be confusing, especially when mixing actions like replacement vs buy/sell iron. If the system displays too many fields or uses ambiguous labels (e.g., “Result”), users lose trust.

Data inconsistency risks between client and server
If the client computes snapshots or totals independently, discrepancies can appear (especially with edits, backdating, or identical timestamps). The backend must be authoritative and deterministic.

Edits/backdating can corrupt historical reporting
Operational reality requires corrections. If an update flow reverses and reposts using “now” instead of the original event time/day, daily reports become inaccurate.

Reconciliation must be possible without exposing complexity upfront
The daily report view must be “scan-first” with minimal data by default, but it must allow expanding any ticket to see ledger before/after impacts for audits and dispute resolution.

5) Domain Model & Source-of-Truth (Core Accounting Semantics)
5.1 Core Principle: Ledger-First, Server-Authoritative

The ledger is the single source of truth. Every business action (“event”) must post one or more ledger entries that represent canonical deltas. All derived states—cash balance, inventory, customer status, company status, daily report timelines—are computed from the ledger, not from UI-calculated snapshots.

Why this matters

Prevents client/server drift.

Makes edits auditable (reverse + repost).

Enables deterministic reporting “as of” an exact boundary.

5.2 Primary Domain Entities (Conceptual)

Event / Transaction (Source)

Represents a user action (Order, Refill, Payment, Adjustment, Expense, etc.).

Has: id, event_type, happened_at, day (business date), created_at, note, created_by, is_reversed, reversed_id.

LedgerEntry (Atomic accounting record)

Represents one atomic delta to one account/unit.

Has: id, source_type, source_id, account, unit, gas_type?, delta, happened_at, day, created_at.

Constraints:

Must be append-only (except reversals).

Must support deterministic ordering and “boundary snapshots”.

Derived Snapshots (Views, not truth)

Inventory snapshot: full/empty by gas type.

Customer balance snapshot: money + cylinders owed/credit by gas type.

Company balance snapshot: money + cylinders owed/credit by gas type.

Day timeline: ordered events with computed before/after impacts.

5.3 Canonical Units & Accounts

Units

money (integer NIS or smallest unit; no floats)

cylinders (integers)

Gas types: 12kg, 48kg

Core accounts (examples)

cash or wallet_cash (distributor cash on hand)

inventory_full_12, inventory_empty_12, inventory_full_48, inventory_empty_48

customer_money_balance (per customer)

customer_cylinder_balance_12/48 (per customer)

company_money_balance (global)

company_cylinder_balance_12/48 (global)

Implementation detail: whether you store separate accounts for “debt” vs “credit” or store a signed net note is internal—but the UI must never expose ambiguous sign semantics. The UI speaks in actions: pay / return.

5.4 Sign Semantics (Internal) vs Action Semantics (UI)

Internally, ledger deltas can be signed:

Positive could mean “increase debt” or “increase asset”, depending on the account.

This is fine—as long as every UI label is derived from context.

UI action semantics should be normalized:

If a cylinder gap exists → display: “Return X x 12kg” or “Return X x 48kg”

If a money gap exists → display: “Pay Y”

Avoid “owes / credit / debt” language in collapsed tickets, because it is cognitively expensive.

5.5 Deterministic “As-Of” Boundaries

To guarantee correct snapshots when timestamps collide:

Ordering key for both reporting and boundaries:

happened_at ascending

created_at ascending

id ascending

Snapshot computation must accept a boundary (not only happened_at):

Include entries strictly before boundary time, plus ties broken by created_at and id.

This ensures:

Two events with identical happened_at produce different “after” snapshots.

Day timeline “before/after” fields remain stable across environments.

5.6 Edit Semantics: Reverse + Repost (Preserving Business Time)

Edits must not rewrite history implicitly.

Rule

When updating an event, the system:

creates a reversal event with equal/opposite ledger entries

posts a new event with updated payload

Both reversal and repost must preserve the original happened_at/day semantics (unless the user explicitly changes the effective time).

This ensures daily reporting remains historically accurate.

5.7 Ticket Rendering Semantics (Report View Contract)

Each event in day_v2 (or equivalent API) should provide enough structured fields to render:

A collapsed ticket (minimal, hide zeros, show ✅ OK when balanced)

An expanded ticket (cash + relevant gas type before/after ledger snapshots)

The frontend should not “guess” too much. It should:

Compute “balanced/unbalanced” from explicit fields (e.g., diff/remaining values).

Render only non-zero actionable deltas.

7) UX Principles & Screen Inventory
7.1 UX Principles (Global)

Scan-first, details on demand

The default view must be scannable in seconds.

Expand reveals “accounting truth” (before/after ledger & inventory), but collapsed shows only what drives action: installed and paid + any non-zero “must pay/return”.

Action verbs, not accounting nouns

Avoid “debt/credit/owes” in collapsed tickets. Use:

Pay (money pending)

Return (cylinders pending)

✅ OK (balanced / nothing to do)

If you want the accounting words, keep them only in expanded (or behind an “Explain” tooltip).

Hide zeros and redundancy

Never show “Result: 0”.

Never show fields that don’t apply to the event type (e.g., “returned” on buy-iron, cylinders on payments).

Prefer “Sold X” / “Bought X” labels over “installed/received” for sell/buy modes.

Consistency across the whole app

Same visual grammar everywhere:

Header: label + counterparty + time

Primary row: Installed/Sold/Bought + Paid

Only if needed: Pay/Return action chips

Expand: before→after blocks for cash and the relevant gas type(s)

Deterministic ordering

Users must trust that “before/after” is correct.

Timeline order must be stable:

happened_at asc → created_at asc → id asc

Ordering (CRITICAL DISTINCTION)

Canonical ordering key (for correctness + boundaries):
happened_at asc, created_at asc, id asc

Display order (for the timeline UI):
The API may return newest-first or oldest-first, but it must include canonical key fields so ordering is deterministic.
Backend uses canonical order for snapshot/boundary math regardless of display order.

Make the “most important” fields visually dominant

Installed and Paid are large, left-to-right, same position in every ticket.

Secondary numbers (total, received, etc.) are smaller or in expanded view.

Collapsed ticket rules

Always show:

7.2.x Status Badge Rules (MUST-MATCH UI)

We have two status concepts:

A) Atomic OK (status_mode="atomic")
Meaning: this event itself has no pending actions (no money gap, no cylinder gap caused by THIS event).
Display label: "✅ OK" (or "OK" if emojis are removed globally).

B) Balance Settled (status_mode="settlement")
Meaning: a settlement-type event (late pay / late return) caused the CUSTOMER'S balance to become fully settled (money + cylinders).
Display label: "✅ Balance settled".

Placement (CRITICAL):
Status must render bottom-right on the SAME ROW as the last left-side line.
If notes/action-pill row exists: status is right-aligned on that row.
If no notes/action-pill row exists: status is right-aligned on the context row.
Status must NOT appear on a new standalone row.

Exception:
Expense (and optionally bank deposit / cash adjust) MUST NOT show ✅ OK or ✅ Balance settled.
(These entries are informational; "OK" creates noise.)

Action pills / notes line-fitting rule (MUST-MATCH UI)
All pill/chip text is single-line only.
If too long, it should:
1) auto-shrink the font (e.g., adjustsFontSizeToFit + minimumFontScale),
2) then ellipsize if still too long.

Expanded ticket rules

Always show cash before→after.

Show only the involved gas types:

Replacement 48 → show only 48 full/empty before→after (+ cash).

Replacement 12 → show only 12 full/empty before→after (+ cash).

Refill with both → show both 12 and 48 before→after (+ cash).

If you track customer/company status snapshots: show those in expanded as a separate block.

8) Data Integrity & Consistency Strategy
8.1 Invariants (Must Always Hold)

Ledger is authoritative

Snapshots shown in UI are derived; backend recalculates after posting.

Deterministic “as-of” boundaries

Any snapshot taken “after event X” must be computed using:

boundary = (happened_at, created_at, id) not happened_at alone.

Edits never rewrite time by accident

Reverse + repost must preserve happened_at/day semantics unless explicitly changed by the user.

No negative physical quantities

Payload validations enforce >= 0 for quantities like installed/received/buy/return/new shells (unless a field is explicitly a delta adjustment).

Inventory adjustments are not purchases

Adjustments are correction-only (reason enum). If purchase-like behavior is needed, it must be a separate event type with ledger semantics.

Sign semantics never leak to the user

UI derives action chips (“Pay/Return”) from net deltas; it does not show “-200 means you owe…” without context.

8.2 Validation Layering

Frontend validation

Required fields present, basic bounds.

Prevent obvious mistakes (empty reason, negative values).

Backend validation (hard enforcement)

All physical quantities >= 0.

Adjustment reason policy enforced.

Business consistency checks where applicable:

Buy iron should not accept refill-style fields.

Refill should reject “new shells” fields.

8.3 Consistency & Drift Controls

Snapshot recompute after each post/update

Backend recalculates relevant balances (customer/company/cash/inventory) and returns them.

Regression tests for tricky cases

Same-timestamp event ordering.

Update backdating correctness.

Reversal correctness (net-zero over reverse+repost vs original).

Database migrations aligned with code

If code depends on ordering fields (e.g., created_at), schema must enforce not-null + indexes.

8.4 Observability & Audit

Every ledger entry has a source_type/source_id.

Reports show the event ID and allow drill-down.

“Deleted” is modeled as reversal rather than hard delete.

10) API Contracts & Versioning
10.1 Contract Principles

Version reports explicitly

Reports are fragile because UI depends on shape and semantics.

Use /reports/day_v2 (already) and freeze semantics.

Do not reuse fields with different meaning

Example: “new shells” must not mean “full cylinders” in refills.

If semantics differ, use different event types or distinct fields.

Server returns computed fields

Each event includes computed deltas or action-friendly fields so UI doesn’t guess.

10.2 Day Timeline Contract (Proposed shape)

GET /reports/day_v2?date=YYYY-MM-DD

Returns:

events[] ordered by (happened_at, created_at, id)

Each event includes:

identity: event_type, source_type, source_id

time: happened_at, created_at, day

label: stable string (e.g., "Sell Full", "Refill")

primary quantities (event-dependent)

computed action fields (UI-ready)

before/after snapshots for expansion

UI-ready computed fields (strongly recommended)

money_to_pay: integer (positive means “Pay X”, zero hides)

cylinders_to_return_12: integer (positive means “Return N x 12kg”, zero hides)

cylinders_to_return_48: integer

is_balanced: boolean (optional but helpful)

This lets the UI render your “✅ OK” rule without recreating accounting logic.

10.3 Write APIs (Commands)

Each “event” has a create/update endpoint. All follow the same pattern:

Accept input payload with explicit happened_at

Backend normalizes day

Backend posts ledger entries

Backend returns:

created/updated event

boundary info (if relevant)

updated snapshots (customer/company/inventory/cash)

Examples:

POST /orders

PUT /orders/{id}

POST /inventory/refill

PUT /inventory/refill/{id}

POST /company/payment

10.4 Versioning Policy

Non-breaking changes: add optional fields, add new event types, add new endpoints.

Breaking changes: renaming fields, changing sign conventions, changing meaning of an event type.

Must bump report version: day_v3.

Maintain at least one previous report version during migration.

10.5 Compatibility Tests (Contract tests)

Add automated tests that assert:

For each event type, the required fields are present.

Ordering key is deterministic.

Computed action fields match ledger-derived truth.


11) Technical Approach & Architecture
11.1 Approach Summary

Event-sourced accounting: business activities (orders/refills/payments/adjustments) are the sources; the ledger is the canonical accounting record derived from them.

Server-authoritative reporting: the backend constructs the daily timeline and returns UI-ready fields (labels + “pay/return” actions + before/after snapshots). The frontend renders; it does not re-implement accounting.

Deterministic snapshot boundaries: every “state after event” is computed with a stable boundary key (happened_at, created_at, id) to avoid same-timestamp ambiguity.

11.2 Key Architecture Decisions

Single daily timeline endpoint

GET /reports/day_v2?date=... returns ordered events with:

label (human-friendly)

minimal “collapsed display data”

expansion payload (before→after snapshots)

optional computed action fields (money_to_pay, cylinders_to_return_12/48, is_balanced)

SmartTicket rendering driven by backend contract

Frontend “SmartTicket” renders by event_type + mode + computed fields.

Goal: remove duplicated sign logic and edge-case divergence.

Posting services are the only ledger writers

Routers validate + call posting services.

Posting returns ledger entries for that source (used to compute boundary snapshots deterministically).

Edits use reverse + repost

Update flow:

reverse existing source entries (same happened_at/day)

create new source event (with updated payload)

post ledger entries

recompute snapshot “after” via boundary

Guardrails are enforced server-side

Non-negative quantities.

Adjustments: reason enum + disallow positive deltas for shrinkage/damage.

“New shells” semantics canonicalized (refill forbids, buy_iron uses).

14) Rollout & Risk Management
14.1 Rollout Strategy

Feature flag the new timeline rendering

Backend can keep /reports/day_v2 stable.

Frontend toggles SmartTicket vs old ticket renderer.

Enables safe staged rollout to internal users first.

Dual-run verification (short window)

For a subset of days, render both old and new summaries (internally) and compare:

totals

“pay/return” actions

cash before/after continuity

Staged deployment

Backend first (contract fields additive).

Frontend next (consumes new fields).

Finally enable feature flag.

14.2 Key Risks & Mitigations

Risk A: Same-timestamp ordering bugs

Mitigation: enforce (happened_at, created_at, id) ordering in DB queries and tests.

Risk B: UI action text wrong (semantic drift)

Mitigation: backend emits computed action fields; frontend only formats strings.

Risk C: Update flows corrupt reporting day

Mitigation: regression tests for “create on day1, edit on day2 → remains on day1”.

Risk D: Adjustment misuse

Mitigation: server-side enforcement + UI copy + restricted reasons.

Risk E: Migration drift

Mitigation: “upgrade head on clean DB” CI step; migration review checklist.

Risk F: Payload size / performance

Mitigation:

keep collapsed payload small

make expansion optional or lazy-load if needed

index day queries

14.3 Operational Monitoring (Minimum viable)

Log posting errors with source_type/source_id.

Track report generation latency.

Add a “data integrity diagnostic” endpoint (optional) for quick checks:

last N ledger entries

snapshots at boundaries
